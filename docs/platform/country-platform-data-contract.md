# Country Platform Data Contract

This contract defines the data required to create a Scientific Intelligence platform for any country. The current MX platform is the reference implementation, but the contract should be generalized for each country.

All user-facing text should be English. Source filenames can preserve original names when needed, but CSV headers, app labels, module text, and exported table labels should be normalized to English.

## Package Layout

Each selected region gets its own package:

```text
public/reports/{region-slug}/
  00_overview/
    manifest.csv
    README.md
  01_final_report/

  05_docs_markdown/

```

For countries that do not use states or municipalities, keep the same conceptual structure but rename fields and app labels. The current MX package contains a few legacy Spanish source filenames; new country packages should prefer English filenames and update the registry helper accordingly.

Example replacements:

- `state` -> `region`, `province`, `department`, or `country unit`.
- `municipality` -> `local_unit`, `district`, `city`, `commune`, or `municipality`.
- `SNI` -> the country's equivalent researcher level, grant category, registry ID, or public researcher classification.

## Region Registry Fields

Every region must be represented in a registry file such as:

```text
src/modules/{country-code}-regions/reportConfig.js
```

Required fields:

| Field | Type | Purpose |
| --- | --- | --- |
| `id` | string | Stable selector key, lowercase and URL-safe. |
| `title` | string | Visible region name. |
| `code` | string | ISO, national, or internal administrative code. |
| `slug` | string | Folder under `public/reports`. |
| `prefix` | string | CSV filename prefix. |
| `accent` | string | UI tone used by charts and badges. |
| `packageDate` | date string | Evidence package date. |
| `mode` | string | Coverage mode, for example `Dense Research`, `Standard`, or `Low Coverage`. |
| `thesis` | string | Investor-facing regional thesis. |
| `tables` | object | Paths generated by the table helper. |

Recommended fields:

| Field | Purpose |
| --- | --- |
| `countryCode` | ISO country code. |
| `adminLevel` | Administrative level used by the selector. |
| `mapFeatureId` | Geodata feature ID for region boundary rendering. |
| `currency` | Currency code for budget bands. |
| `licenseNotes` | Short source/license description. |
| `coverageWarnings` | Known gaps in market or science coverage. |

## Derived KPI Guidelines

When a KPI is not present in source files, create it as a documented derived metric.

Recommended derived metrics:

- `market_entry_score`: demand scale, sector concentration, local concentration, company size diversity, and contactability.
- `science_readiness_score`: evidence strength, official signals, institution count, output count, and identity resolution confidence.
- `investment_readiness_score`: weighted combination of market entry, science readiness, time to revenue, budget feasibility, and risk pressure.
- `proof_feasibility_score`: 90-day deliverable clarity, pilot client availability, technical gate clarity, commercial gate clarity, and owner completeness.
- `international_investment_fit`: exportability, IP potential, foreign partner fit, applied science depth, and buyer urgency.

Every derived metric needs:

- Formula or scoring rule.
- Input columns.
- Weight assumptions.
- Missing data behavior.
- Human-readable interpretation.
- Date generated.

## Privacy And Anonymization

People:

- Replace public researcher names with stable anonymized IDs unless explicitly approved.
- Include public researcher levels or registry numbers when allowed, for example SNI number or national equivalent.
- Remove personal emails, phones, and private addresses from public downloads.
- Keep source URLs and public registry references where legally allowed.
- Maintain a private crosswalk outside the public repo if names must be recoverable internally.

Companies:

- Named pilot accounts can be shown when the source is public and the project approves it.
- If contact details are scraped or low-confidence, publish contactability status instead of raw phone/email.
- Include `source`, `source_date`, and `contact_privacy_status`.

Institutions:

- Institution names are generally acceptable when sourced from public registries or official sites.
- Include source URLs for official signals and commercialization readiness.

## Map Data Requirements

For every region:

- Boundary geometry for the selected region.
- Boundary geometry or centroids for local units shown in Market Intelligence.
- Stable ID mapping between CSV and geometry.
- Coordinate reference system documented in the manifest.
- License documented in the manifest.

For choropleths:

- `geometry_id`
- `local_unit_code`
- `local_unit_name`
- metric field

For points:

- `latitude`
- `longitude`
- point label
- size metric
- color metric

Do not use random coordinates, hand-drawn silhouettes, or unlabeled bubbles in final deployment.

## Validation Checklist

- All required report folders exist for every allowed region.
- Every registry table path resolves or is intentionally `null`.
- CSV files parse with headers and at least one row where coverage is claimed.
- Numeric columns contain parseable numbers.
- Currency bands include a currency code.
- Map IDs match geodata.
- Researcher data is anonymized or approved.
- User-facing strings are English.
- All module controls change visible data.
- All visible data surfaces include a raw download.
- `npm run build` passes.
- The selected deployment target starts successfully.
- Protected routes redirect when logged out.
- Health, status, feedback, legal, and contributor documentation work; protected operations pass authorization tests.
